<task type="auto">
  <name>task 1: Implement /openpaul:complete-milestone command</name>
  <files>src/commands/complete-milestone.ts</files>
  <action>
Create complete-milestone command following patterns from remove-phase.ts and unify.ts:

1. Define Zod args schema:
   - name: optional string (milestone name, defaults to active milestone)
   - confirm: optional boolean (skip confirmation prompt)
   - verbose: optional boolean (show detailed breakdown)

2. Command behavior (per CONTEXT.md decisions):
   - If no name provided, use active milestone
   - Show summary of what will be archived before proceeding
   - Prompt for confirmation unless --confirm flag
   
3. Implementation flow:
   a. Load milestone via MilestoneManager.getMilestone() or getActiveMilestone()
   b. Calculate metrics:
      - Phases completed / total
      - Plans completed / total (from phase SUMMARYs)
      - Total execution time (from STATE.md metrics)
      - Requirements addressed (from REQUIREMENTS.md traceability)
   c. Display summary:
      ```
      ## Completing Milestone: v1.1 Full Command Implementation
      
      **Phases:** 7/7 complete (100%)
      **Plans:** 50/50 complete
      **Execution Time:** 2.5 hours
      **Requirements:** 22/22 addressed
      
      This will:
      - Archive milestone to .planning/MILESTONE-ARCHIVE.md
      - Collapse phases to <details> section in ROADMAP.md
      - Update progress tracking
      
      Confirm? (yes/no)
      ```
   d. If confirmed, call MilestoneManager.completeMilestone()
   e. Format success output with archive location

4. Archive entry format (per CONTEXT.md - summary + metrics):
   - Name, dates, phases
   - Plans completed, total execution time
   - Requirements addressed
   - Appended to MILESTONE-ARCHIVE.md

5. Error handling:
   - Milestone not found → error with available milestones
   - Milestone already complete → error
   - Phases not complete → warning with incomplete list

Register command in src/index.ts as `paulCompleteMilestone`.
