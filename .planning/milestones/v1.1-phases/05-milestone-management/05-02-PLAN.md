<task type="auto">
  <name>task 1: Implement /openpaul:milestone command</name>
  <files>src/commands/milestone.ts</files>
  <action>
Create milestone command following patterns from add-phase.ts and plan.ts:

1. Define Zod args schema:
   - name: optional string (milestone name)
   - scope: optional string (milestone scope/description)
   - phases: optional string (comma-separated phase numbers, e.g., "3,4,5,6")
   - theme: optional string (optional theme/slogan)
   - updateState: optional boolean (whether to update STATE.md)

2. Command behavior (per CONTEXT.md decisions):
   - **Hybrid mode**: Interactive by default, CLI args for scripting
   - If no args provided, prompt user for name, scope, phases
   - If args provided, use them directly (scripting mode)
   
3. Implementation flow:
   a. Load ROADMAP.md via RoadmapManager
   b. Validate phase numbers exist in roadmap
   c. Call MilestoneManager.createMilestone(name, scope, phases, theme)
   d. Format output: show milestone created with summary
   e. **Prompt user**: "Update STATE.md to track this milestone?" (unless --updateState flag)
   f. If yes, update STATE.md current_milestone field

4. Output format (per CONTEXT.md - header + bullet list):
   ```
   ## Milestone: v1.1 Full Command Implementation
   
   **Scope:** Implement all remaining PAUL commands (22 total) for complete structured development workflow
   
   **Phases:** 3, 4, 5, 6, 7, 8, 9
   
   **Theme:** Full Feature Set (optional)
   
   **Status:** Planned
   ```

5. Error handling:
   - Invalid phase numbers → list valid phases
   - Duplicate milestone name → error with suggestion
   - File write failures → return formatted error message

Register command in src/index.ts as `paulMilestone`.
